## 섹션 A. 왜 만들었나 — 문제의식 (약 2분 · 슬라이드 1~2장)

[핵심 메시지] **반복되는 세팅, 흐려지는 표준, 폐쇄망 제약 — 이 세 가지를 풀려고 만들었다.**

[말할 내용]
> 새 React 프로젝트나 새 화면을 시작할 때마다 우리는 비슷한 일을 반복합니다. 폴더 구조 잡고, 공통 컴포넌트 다시 만들고, API 통신·상태관리·라우팅을 매번 똑같이 세팅하죠.
> 문제는 두 가지입니다. 첫째, 이 반복이 시간을 잡아먹습니다. 둘째, 사람마다 조금씩 다르게 짜다 보니 **시간이 지나면 팀의 표준이 흐려집니다.**
> 게다가 저희 환경은 **폐쇄망**입니다. 외부 ChatGPT나 Copilot처럼 코드를 외부 클라우드로 보내는 도구는 쓸 수 없습니다.
> 그래서 저희는 두 개를 만들었습니다 — 표준을 담은 **토대(scaffold)**, 그리고 그 위에서 폐쇄망 안에서 동작하는 **AI 어시스턴트(Axiom)** 입니다.

[슬라이드로 뽑을 거리]
- 반복 세팅 / 표준 흐려짐 / 폐쇄망 제약 — 3개 페인 포인트
- "그래서 만든 두 가지" 한 장 전환 슬라이드

## 섹션 B. 큰 그림 — 두 개의 축 (약 1분 30초 · 슬라이드 1장)

[핵심 메시지] **scaffold는 '토대', Axiom은 그 위에서 일하는 'AI 어시스턴트'.**

[말할 내용]
> 전체 구조는 두 층입니다.
> 아래층은 **react-app-scaffold** — React와 TypeScript로 만든, 실무에서 바로 쓰는 컴포넌트·유틸리티·예제를 담은 스타터 프로젝트입니다. 우리 팀의 표준이 코드와 문서로 들어 있는 토대죠.
> 위층은 **Axiom AI** — 그 scaffold를 이해하는("Scaffold-Aware") VSCode AI 어시스턴트입니다. scaffold의 규칙과 컴포넌트를 RAG로 읽어서, 그 표준대로 코드를 만들어 줍니다.
> 순서대로 — 먼저 토대인 scaffold, 그 다음 그 위의 Axiom 순으로 보겠습니다.

[슬라이드로 뽑을 거리]
- 2층 다이어그램: (아래) react-app-scaffold = 표준·컴포넌트·유틸 / (위) Axiom AI = RAG 기반 코드 생성

## 섹션 C. react-app-scaffold (약 5~6분 · 슬라이드 다수)

### C-1. 한마디로 무엇인가 (슬라이드 1장)

[핵심 메시지] **"바로 가져다 쓰는" React·TypeScript 스타터 — 초기 세팅과 개발을 빠르게.**

[말할 내용]
> react-app-scaffold는 React.js와 TypeScript 기반으로, 실무에서 바로 사용할 수 있는 **UI 컴포넌트·유틸리티·예제 코드**를 제공하는 스타터 프로젝트입니다.
> 목적은 단순합니다 — 프로젝트 초기 세팅과 반복 개발을 없애서, 개발자가 비즈니스 로직에만 집중하게 하는 것입니다.

[슬라이드로 뽑을 거리]
- 정의 한 줄 + 주요 목적 4가지(UI 컴포넌트 / 유틸리티 / 인터랙티브 데모 / 코드 예제)

### C-2. 무엇을 제공하나 — 컴포넌트 · 유틸리티 (슬라이드 2~3장)

[핵심 메시지] **shadcn/ui를 래핑한 자체 컴포넌트와, 전역 유틸리티를 한 곳에서.**

[말할 내용]
> 제공물은 크게 둘입니다.
> 하나는 **UI 컴포넌트** — `shadcn/ui`를 래핑한 자체 컴포넌트를 `@/shared/ui` 한 곳에서 import해 씁니다. Accordion 같은 컴포넌트들이 들어 있고, 필요에 따라 계속 추가됩니다.
> 다른 하나는 **유틸리티** — 전역 객체 `$util`, `$router`, `$ui` 와 커스텀 훅으로 제공됩니다. API 통신, 날짜 포맷, 배열·객체 조작, 데이터 변환 같은 자주 쓰는 로직을 직접 구현할 필요 없이 바로 호출할 수 있습니다.
> 그리고 각 컴포넌트마다 **직접 조작해보는 인터랙티브 데모와 복사해서 바로 쓰는 코드 예제**를 함께 줍니다. 학습 곡선을 낮추는 부분입니다.

[슬라이드로 뽑을 거리]
- UI 컴포넌트: shadcn/ui 래핑 · `@/shared/ui` 단일 진입점
- 유틸리티: `$util` · `$router` · `$ui` + 커스텀 훅 (예: API/날짜/배열·객체/변환)
- 인터랙티브 데모 + 코드 예제 (DX)

### C-3. 디자인 시스템 — 토큰과 퍼블리싱 (슬라이드 1~2장)

[핵심 메시지] **디자인 값을 JSON 토큰으로 일원 관리하고, 퍼블리셔 전용 작업 공간을 따로 둔다.**

[말할 내용]
> scaffold는 **Style Dictionary 기반 디자인 토큰 시스템**을 내장합니다. 색상·타이포그래피·그림자 같은 디자인 값을 JSON으로 한 곳에서 관리하고, 자동으로 CSS 변수로 만들어 줍니다. 토큰은 2계층 구조로 관리됩니다.
> 또 **퍼블리셔 전용 작업 공간**(`src/publishing/`)을 별도로 둬서, 퍼블리셔가 디자인 토큰과 Tailwind로 React 컴포넌트를 자유롭게 작성하고 Storybook으로 팀에 공유할 수 있습니다. 개발자와 퍼블리셔의 협업 경계를 깔끔하게 나눈 거죠.

[슬라이드로 뽑을 거리]
- 디자인 토큰: Style Dictionary · JSON 일원 관리 → CSS 변수 자동 생성 · 2계층
- 퍼블리싱 작업 공간: `src/publishing/` + Storybook 공유

### C-4. 설계 철학과 장점 (슬라이드 2~3장)

[핵심 메시지] **헤드리스 + 도메인 기반 구조 + 타입 안정성 + 접근성 — '바꾸기 쉬운' 토대.**

[말할 내용]
> 설계에서 강조하고 싶은 게 몇 가지 있습니다.
> - **헤드리스 컴포넌트 방식**: MUI나 AntD처럼 완성된 UI를 강제하지 않고, 로직·상태만 제공하고 UI는 우리가 자유롭게 입힙니다. 그래서 프로젝트 상황에 맞게 수정·확장이 쉽습니다. 스타일도 Tailwind를 기본으로 쓰되, Sass·Emotion·CSS Modules 어떤 것이든 붙일 수 있습니다.
> - **도메인 기반 구조(DDD)**: 기능·업무별로 도메인을 분리해서 복잡도와 충돌을 줄이고, 유지보수성과 확장성을 높였습니다.
> - **타입 안정성**: 모든 컴포넌트가 TypeScript로 작성돼 props·반환값 타입이 명확하고, VSCode 자동완성과 타입 체크가 잘 됩니다.
> - **접근성**: shadcn/ui(Radix) 기반이라 키보드 조작과 스크린 리더(ARIA)를 기본 지원합니다.
> - **상태관리·성능**: 서버 상태는 TanStack Query, 클라이언트 상태는 Zustand, 통신은 Axios 공통 클라이언트로 표준화했고, 코드 스플리팅과 Vite HMR로 빠른 개발·실행을 챙겼습니다.

[슬라이드로 뽑을 거리]
- 헤드리스 컴포넌트 / 유연한 스타일링
- 도메인 기반 구조(DDD) · 모듈화
- 타입 안정성 · 접근성
- 상태관리(TanStack Query · Zustand · Axios) · 성능(코드 스플리팅 · Vite HMR)

### C-5. 기반 기술 스택 (슬라이드 1장)

[핵심 메시지] **최신 표준 스택 위에 세웠다.**

[말할 내용]
> 기반 기술은 최신 스택입니다. **React 19, TypeScript, Vite** 위에, UI는 **Tailwind CSS v4 + shadcn/ui(Radix)**, 테스트는 **Vitest + Playwright + Storybook**, 품질 도구로 **ESLint + Prettier** 를 씁니다. 라우팅은 React Router, 데이터 페칭은 TanStack Query + Axios 입니다.

[슬라이드로 뽑을 거리]
- 스택 로고 그리드: React 19 / TypeScript / Vite / Tailwind v4 / shadcn/ui / TanStack Query / Vitest·Playwright·Storybook / ESLint·Prettier

## 섹션 D. Axiom AI (약 5~6분 · 슬라이드 다수)

### D-1. 한마디로 무엇인가 (슬라이드 1장)

[핵심 메시지] **scaffold를 이해하는, 폐쇄망에서 도는 VSCode AI 개발 어시스턴트.**

[말할 내용]
> Axiom AI는 한 줄로 **"Scaffold-Aware Developer Assistant"** — react-app-scaffold를 이해하는 프론트엔드 개발 AI 어시스턴트입니다. VSCode 확장으로 설치되고, 사이드바에서 채팅으로 대화하며 코드를 만들고 고칩니다.
> 핵심 차별점은 **폐쇄망 sLLM 연동**입니다. 외부 클라우드에 코드를 보내지 않고, 사내에 띄운 로컬 LLM만으로 동작합니다.

[슬라이드로 뽑을 거리]
- 정의 + VSCode 사이드바 채팅 UI 캡처
- "폐쇄망 / 로컬 LLM" 배지

### D-2. 왜 폐쇄망·로컬인가 (슬라이드 1장)

[핵심 메시지] **코드를 밖으로 내보내지 않는다 — 보안과 도입 장벽을 동시에 푼다.**

[말할 내용]
> 가장 큰 이유는 **보안**입니다. 사내 코드가 외부로 나가지 않습니다.
> LLM 백엔드는 OpenAI 호환 API나 Ollama를 연동할 수 있고, 모델은 사내 환경에 맞춰 교체합니다(기본값은 qwen3-coder 계열). 즉, 회사가 가진 GPU 자원과 모델에 맞춰 유연하게 붙일 수 있습니다.

[슬라이드로 뽑을 거리]
- 데이터 흐름도: 코드 ↔ 사내 LLM (외부 클라우드 차단 표시)
- 지원 백엔드: OpenAI 호환 / Ollama · 교체 가능한 모델

### D-3. 핵심 — RAG로 '우리 표준'을 알고 생성한다 (슬라이드 2장)

[핵심 메시지] **scaffold의 규칙·컴포넌트·유틸을 RAG로 검색해, 그 표준대로 코드를 만든다.**

[말할 내용]
> Axiom의 진짜 핵심은 여기입니다. 일반 AI는 학습된 일반 지식으로 코드를 짭니다. 그래서 "우리 팀 방식"은 모릅니다.
> Axiom은 **RAG**를 씁니다. scaffold의 규칙과 컴포넌트·유틸리티 사용법을 정리한 마크다운 지식(corpus)을 검색해서, 그 근거를 바탕으로 코드를 생성합니다. 그래서 "그냥 동작하는 코드"가 아니라 **"우리 scaffold 방식에 맞는 코드"** 가 나옵니다.
> 그리고 이 지식베이스는 닫혀 있지 않습니다. 팀이 추가 RAG 폴더나 개별 문서를 지정하면 그대로 인덱스에 포함됩니다. **규칙이 바뀌면 코드가 아니라 문서를 고치면 됩니다.**

[슬라이드로 뽑을 거리]
- RAG 흐름: 질문 → corpus(.md) 검색 → 근거와 함께 생성
- "Scaffold-Aware": @/shared/ui 컴포넌트·$util 유틸을 알고 사용
- 확장 가능한 지식: 사용자 RAG 폴더/문서 추가

### D-4. 안전한 코드 적용 (슬라이드 2장)

[핵심 메시지] **파일을 통째로 갈아엎지 않는다 — 바뀐 부분만, 검토 가능한 형태로.**

[말할 내용]
> AI 코드 도구의 가장 큰 불안은 "멀쩡한 파일을 망가뜨리는 것"입니다. Axiom은 이걸 적용 방식으로 막습니다.
> - **부분 수정(multi-patch)**: 한 번 응답으로 여러 곳을 부분만 정확히 고칩니다. 통째로 다시 쓰지 않습니다.
> - **라인 단위 편집(line edit)**: 바뀐 줄만 지정해 고쳐서, 출력도 적고 변경 범위도 작게 유지합니다.
> - **안전장치**: 고쳐야 할 위치를 못 찾으면 억지로 적용하는 대신, 실제 코드를 다시 확인해 한 번 더 시도하고, 그래도 모호하면 사용자에게 선택을 넘깁니다. 선택 영역 밖으로 변경이 번지는 것도 막습니다.
> - **오프라인 폴백**: LLM 서버가 응답하지 않으면, scaffold 기반의 빈 스텁이라도 안전하게 돌려줘서 작업이 끊기지 않게 합니다.
> 결과적으로 모든 변경이 **사람이 검토하고 되돌릴 수 있는** diff 형태로 들어옵니다.

[슬라이드로 뽑을 거리]
- multi-patch / line-edit diff 예시(before→after)
- 안전장치: grounded retry · 변경 범위 가드 · 오프라인 스텁 폴백

### D-5. 작은 모델에서도 잘 돌게 (슬라이드 1장)

[핵심 메시지] **폐쇄망의 제한된 sLLM에 맞춰 프롬프트를 최적화했다.**

[말할 내용]
> 폐쇄망에서는 거대 모델을 쓰기 어렵습니다. 그래서 Axiom은 **제한된 sLLM에서도 동작하도록** 프롬프트를 다듬습니다.
> 단순 질의응답에는 편집 지시문을 빼서 가볍게 보내고, 컨텍스트가 클 때는 RAG 분량을 동적으로 줄여 전체 프롬프트가 무한정 커지지 않게 합니다. 또 Qwen 계열의 불필요한 추론(thinking) 토큰을 꺼서 빈 응답·낭비를 줄입니다. 작은 모델을 똑똑하게 쓰는 엔지니어링이 들어가 있습니다.

[슬라이드로 뽑을 거리]
- 프롬프트 다이어트: Q&A 게이팅 · 적응형 예산 · thinking off
- "작은 모델 + 정교한 프롬프트" 컨셉 도식

### D-6. 사용 흐름 (슬라이드 1장)

[핵심 메시지] **사이드바에서 대화하면 끝 — 자연스러운 개발 흐름 안에 들어온다.**

[말할 내용]

## 섹션 E. 둘이 어떻게 이어지나 (약 1분 30초 · 슬라이드 1장)

[핵심 메시지] **scaffold가 표준을 제공하고, Axiom이 그 표준을 읽어 안전하게 코드로 만든다.**

[말할 내용]
> 정리하면 두 도구는 하나의 흐름입니다.
> **react-app-scaffold** 가 "우리 팀의 표준 — 컴포넌트·유틸·규칙"을 코드와 문서로 제공하고,
> **Axiom AI** 가 그 표준을 RAG로 읽어서, 개발자의 의도 한 줄을 그 표준에 맞는 코드로 만들어, 안전한 diff로 적용합니다.
> 토대가 좋아질수록 AI도 똑똑해지는, 서로를 키우는 구조입니다.

[슬라이드로 뽑을 거리]
- 의도 → (Axiom) RAG 검색 → 생성 → 안전 적용 → 코드, 한 줄 파이프라인
- scaffold ↔ Axiom 상호 강화 루프

## 부록 — 예상 Q&A (참고용)

- **Q. 외부 ChatGPT·Copilot과 뭐가 다른가요?**
  → 폐쇄망입니다. 코드를 외부로 보내지 않고 사내 로컬 LLM으로 동작합니다. 또 일반 지식이 아니라 **우리 scaffold 표준을 RAG로 읽어** 코드를 만들고, 변경을 통째가 아니라 diff 단위로 안전하게 적용합니다.

- **Q. 우리 팀 규칙이 바뀌면요?**
  → 코드가 아니라 RAG 문서(.md)만 고치면 됩니다. 다음 생성부터 반영됩니다. 추가 지식 폴더/문서도 지정해 넣을 수 있습니다.

- **Q. AI가 파일을 망가뜨리지 않나요?**
  → 부분 patch·라인 단위 편집으로 바뀐 곳만 고치고, 위치를 못 찾으면 억지 적용 대신 재확인/사용자 선택으로 넘깁니다. 모든 변경이 검토·되돌리기 가능한 diff입니다.

- **Q. 어떤 모델을 쓰나요? 사양은?**
  → OpenAI 호환 또는 Ollama로 사내 sLLM을 연동합니다(기본 qwen3-coder 계열). 모델은 사이트의 GPU 환경에 맞춰 교체 가능합니다.

- **Q. scaffold만 써도 되나요? Axiom만 써도 되나요?**
  → scaffold는 단독 스타터로도 충분히 쓰입니다. Axiom은 그 위에서 생산성을 더 끌어올리는 층이라, 함께 쓸 때 효과가 가장 큽니다.

- **Q. 지금 어디까지 됐나요?**
  → *(현재 진척·적용 대상 프로젝트를 발표자가 채워주세요.)*
